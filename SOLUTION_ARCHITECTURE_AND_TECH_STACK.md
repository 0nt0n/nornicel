# Описание решения и технологический стек проекта

Данный документ представляет собой развернутое описание архитектуры, бизнес-логики и используемого стека технологий в проекте (GraphRAG-система для горно-металлургии).

## 1. Концепция и описание решения

Система представляет собой **GraphRAG** (Retrieval-Augmented Generation на основе графа знаний), спроектированную специально для анализа документов в горно-металлургической отрасли. 

**Основная цель:** Позволить пользователю загружать разнородные неструктурированные документы (статьи, отчеты, презентации) на русском и английском языках, автоматически извлекать из них сущности, связи и числовые ограничения, и задавать вопросы на естественном языке, получая точные ответы со ссылками на первоисточники.

### Ключевые этапы (Пайплайн):
1. **Инджест (Ingestion):** Чтение файлов различных форматов, разбиение на логические чанки, определение языка.
2. **Экстракция (Extraction):** Обработка каждого чанка с помощью LLM (через Structured Output) для извлечения графа по заранее заданному контракту (сущности, такие как Материал, Процесс, Оборудование, и связи между ними). 
3. **Построение графа (Graph Building):** Сохранение извлеченных данных в графовую базу данных с дедупликацией сущностей. Попутно создаются векторные индексы по текстовому описанию сущностей.
4. **Поиск и синтез (Retrieval & Synthesis):** Роутинг вопроса пользователя на интент и фильтры, поиск релевантного подграфа (векторный поиск + Cypher-шаблоны) и генерация итогового ответа со ссылками на источники.

## 2. Архитектура и компоненты

### 2.1 Язык и основные библиотеки
* **Язык программирования:** Python 3.10+
* **Управление окружением:** `venv`
* **Валидация контрактов:** `pydantic` (версии 2.6+) — используется для жесткого контроля схемы данных (онтологии), которую возвращает LLM. Описывает сущности, связи и форматы выгрузки JSON-LD.
* **Работа с переменными окружения:** `python-dotenv`

### 2.2 LLM и AI-инструментарий
* **Провайдер:** Yandex AI Studio (в контуре заказчика для информационной безопасности).
* **Клиент:** `openai` (>=1.40). Yandex AI Studio предоставляет OpenAI-совместимый API, что позволяет использовать стандартные и проверенные библиотеки для взаимодействия (например, получение Structured Output).
* **Модели:**
  * Извлечение сущностей и синтез ответа: `yandexgpt` (или тяжелые модели вроде `qwen3-235b-a22b-fp8`, доступные в Yandex Cloud).
  * Быстрый роутинг запросов (определение интента пользователя): `yandexgpt-lite`.
* **Эмбеддинги (векторизация):**
  * Облачный вариант: Yandex Embeddings.
  * Локальный вариант: модель E5 (используется опционально через библиотеку `sentence-transformers` для полностью офлайн работы).

### 2.3 Графовая база данных и поиск
* **СУБД:** **Neo4j** (версия 5.20+ / 5.26-community). Идеально подходит для работы со связями и поддерживает векторные и полнотекстовые индексы из коробки.
* **Плагины Neo4j:** `APOC` (Awesome Procedures on Cypher) — расширение стандартных возможностей Cypher для более сложной логики и импорта/экспорта.
* **Драйвер:** `neo4j` (Python-клиент).
* **Поисковые движки (встроенные в Neo4j):** 
  * *Vector index* — семантический поиск по смыслу.
  * *Lucene fulltext index* — точный лексический поиск (критически важно для поиска специфичных аббревиатур или металлургических терминов, например "Cu-EW"). В проекте применяется гибридный RRF (Reciprocal Rank Fusion) поиск.

### 2.4 Парсинг корпуса (Ingestion Stack)
Обработка "грязных" и неструктурированных данных — важнейший этап проекта.
* **PDF-документы:** `PyMuPDF` (версия >=1.24, импортируется как `fitz`). Мощный, быстрый инструмент для извлечения текста из PDF.
* **Презентации (.pptx):** `python-pptx` (>=0.6.23).
* **Текстовые документы Word (.docx):** `python-docx` (>=1.1).
* **Определение языка текста:** `langdetect` (>=1.0.9). Используется для маршрутизации и промптинга LLM на правильном языке.
* **Определение кодировок (для txt/csv):** `chardet` (>=5.2.0).
* **Работа с Rich Text Format (.rtf):** `striprtf` (>=0.0.26).

### 2.5 Пользовательский интерфейс (Frontend)
Вместо написания сложного React/Vue фронтенда, используется легковесный стек для Data Science:
* **Интерфейс:** `streamlit` (>=1.33). Позволяет строить интерактивные дашборды, чат-интерфейс, отображать колонки, слайдеры и т.д. на чистом Python.
* **Визуализация графа:** `pyvis` (>=0.3.2). Генерирует интерактивные HTML-виджеты с графами, которые встраиваются в Streamlit. Позволяет пользователю "пощупать" связи между извлеченными материалами и процессами.
* **Прогресс-бары:** `tqdm` (используется для CLI-интерфейса и скриптов массовой обработки).

### 2.6 Развертывание (DevOps и Инфраструктура)
* **Контейнеризация:** `Docker`. Сборка приложения описана в `Dockerfile`.
* **Оркестрация:** `Docker Compose` (`docker-compose.yml`, `docker-compose.prod.yml`, `docker-compose.tunnel.yml`). Позволяет поднимать сразу всё окружение одной командой (база данных Neo4j + Python-приложение).
* **Публичный доступ без "белых IP":** `cloudflared` (Cloudflare quick tunnel). Используется в `docker-compose.tunnel.yml`. Позволяет пробросить локальный порт 8501 (Streamlit) наружу в интернет и получить публичную HTTPS ссылку без регистрации и привязки доменов (идеально для демо).

## 3. Архитектурные паттерны и решения (Highlights)

1. **Structured Output (Контракт):** Вся онтология зашита в коде (`schema/ontology.py`). LLM возвращает не сырой текст, а строгий JSON, который валидируется Pydantic-моделями перед сохранением в базу.
2. **Чекпоинты (Robust Extraction):** Парсинг и экстракция разделены. При извлечении графа из текста результат сохраняется после *каждого* чанка в `data/processed/*.json`. Если скрипт упадет, прогресс не потеряется.
3. **Безопасный RAG (Cypher Templates):** В отличие от классического Text2Cypher, где LLM пишет запросы к базе напрямую (что чревато инъекциями и галлюцинациями), здесь LLM занимается только *маршрутизацией* (определением интента и фильтров). Сформированные параметры подставляются в безопасные, написанные руками шаблоны Cypher (`src/graph/queries.py`).
4. **Совместимость с FAIR-принципами:** Извлеченные графовые данные могут быть экспортированы в формат **JSON-LD** (`scripts/export_jsonld.py`), который является стандартной сериализацией RDF, что позволяет интегрироваться с корпоративными Semantic Web системами.
